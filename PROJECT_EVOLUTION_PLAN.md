# AutoCADMindAI 项目进化规划（方案 C）

## 1. 项目目标（重新统一）

### 核心目标
在不改变总体技术路线（**AutoCAD C# 插件 + Python AI 能力层**）的前提下，持续降低用户操作复杂度，达到：

1. 用户尽量不离开 AutoCAD 主界面
2. 用户不需要理解 Python / Bridge / 脚本细节
3. 用户只需“打开 AI、发送、停止”三类简单动作

### 非目标
- 不重写 AI 能力层
- 不放弃当前 Python UI 与业务能力
- 不做大规模架构推翻

---

## 2. 当前状态总结（截至当前）

### 已完成
- Python 端：
  - AI 对话与命令执行流程
  - 停止逻辑增强（可取消 CAD 当前命令）
  - 基础桥接服务（`ipc_bridge.py`）
- C# 插件端：
  - `NETLOAD` 后可用命令：`AIMIND`, `AICHAT`, `AISTOP`, `AISTART`, `AIPING`
  - 可调用 Python 侧 Bridge
- 工程化：
  - `build_plugin.bat` 一键编译
  - `package_release.bat` 一键打包
  - `start.bat` + `bridge_start.log` 基础启动与日志

### 暴露问题
1. 普通用户路径过长：编译、打包、NETLOAD、命令组合
2. 用户在 AutoCAD 与外部窗口之间切换仍频繁
3. 启动链路（Python/依赖/Bridge）容易出现环境问题
4. 调试信息技术化，普通用户难以理解

---

## 3. 目标产品形态（保持方案 C）

## 最终形态
- AutoCAD 内置 **Palette 停靠面板** 为主交互入口（类似原生命令面板）
- Python 作为后台 AI 引擎服务
- AutoCAD 用户默认不接触 Python 控制台与 bat 脚本

## 用户视角
- 打开 AI（按钮）
- 输入需求并发送
- 停止当前处理

只有这三件事。

---

## 4. 进化路线图

## 阶段 V1（稳定可用，短期）
目标：先把复杂度从“多命令”降到“单入口”。

### V1-1 交互收敛
- 对用户只保留 `AIMIND`（单入口）
- `AIMIND` 内部自动执行：
  - Bridge 探活
  - 如未启动则后台拉起 Python
  - 唤起 AI 窗口
- `AISTART/AIPING` 仅保留开发者模式使用

### V1-2 启动与容错
- 统一启动日志到 `bridge_start.log`
- 启动失败时输出“人话提示”+建议动作
- 避免 AutoCAD 命令被阻塞（减少同步长等待）

### V1-3 发布标准化
- 固定发布目录结构（`dist/AutoCADMindAI`）
- 打包后自动输出版本号、时间戳、关键文件检查

---

## 阶段 V2（AutoCAD 内原生体验，中期）
目标：减少窗口切换，回到 AutoCAD 内完成交互。

### V2-1 Palette 面板（重点）
- C# 插件创建可停靠面板（PaletteSet）
- 面板中提供：
  - 输入框
  - 发送按钮
  - 停止按钮
  - 状态显示

### V2-2 Python UI 保留策略
- Python UI 继续保留（调试/高级模式）
- 普通用户默认通过 AutoCAD 面板交互

### V2-3 命令可视化
- 将 AI 响应区分为：
  - 对话回复
  - 即将执行命令（可选确认）

---

## 阶段 V3（可交付部署，长期）
目标：降低安装与运维门槛。

### V3-1 安装器化
- 制作安装器（Setup）
- 自动完成：
  - 插件文件部署
  - Python runtime 检查/安装
  - 必要配置生成

### V3-2 自动加载
- 尽量实现 AutoCAD 启动后插件自动可用（减少手工 NETLOAD）

### V3-3 一键诊断
- 面板提供“导出诊断包”
- 自动打包日志、版本、配置

---

## 5. 架构边界与职责

### C# 插件职责（前台）
- AutoCAD 内 UI（Ribbon/Palette/命令）
- 用户交互入口
- 将请求发送给 Python 服务
- 执行结果反馈（状态、提示）

### Python 职责（后台）
- AI 模型调用
- 多轮上下文管理
- 命令解析与意图判定
- 停止与会话控制

### 通信约定（建议固定）
```json
{
  "request_id": 123,
  "intent": "chat|command",
  "response": "...",
  "commands": ["CIRCLE"]
}
```

---

## 6. 质量门槛（每次迭代验收）

1. 关键路径不超过 2 步：
   - 打开 AI -> 发送
2. 停止动作 1 秒内可反馈
3. 启动失败必须给可执行提示，不只报技术错误
4. 所有调试日志可定位问题来源（C# / Python / Bridge）

## 9. 更新机制（每次完成并测试成功后立即更新本文档）

### 9.1 状态枚举
- `待办`：未开始
- `进行中`：正在开发或联调
- `已完成`：开发完成且测试通过，达到任务目标
- `阻塞`：存在外部依赖或关键问题，需说明阻塞原因

### 9.2 更新时机
1. 任务开始时：将状态改为 `进行中`
2. 任务开发完成后：先执行对应“测试与通过判定”
3. 测试全部通过后：将状态改为 `已完成`，并补充测试日期与结果摘要
4. 若测试失败：保持 `进行中` 或改为 `阻塞`，并记录失败原因与下一步

### 9.3 更新模板（直接复制到任务下方）
- 完成日期：YYYY-MM-DD
- 代码变更：`文件A`、`文件B`
- 测试结果：通过/失败（用例数：X，成功：Y，失败：Z）
- 结论：是否达到既定目标（是/否）
- 备注：风险与后续动作
