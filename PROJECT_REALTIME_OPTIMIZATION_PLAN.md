# AutoCADMindAI 实时优化方案文档（V1）

## 1. 目标
在现有“可用”基础上，形成可持续迭代的实时智能绘图体系：

1. 复杂图形请求可追踪（每次请求都有状态与日志）
2. 复杂图形结果可验证（语义正确性 > 仅命令可执行）
3. 全链路可回放（请求、分析、生成、执行、重试）
4. 保持大模型主导（不写死固定图形模板）

---

## 2. 当前问题

- 执行成功但语义偏差（如椅子画成桌子）
- 模型偶发返回空/损坏JSON，导致链路降级为 chat
- 自动布局有时破坏原始几何关系（标注场景已部分修复）
- 日志缺少统一结构，复杂问题定位成本高

---

## 3. 总体架构（实时链路）

请求生命周期状态（State Machine）：

- `RECEIVED`：接收用户请求
- `ANALYZED`：完成意图/需求分析
- `GENERATED`：生成绘图命令
- `VALIDATED`：命令清洗 + 语义质量检查
- `EXECUTING`：下发 AutoCAD 执行
- `EXECUTED`：执行完成
- `RETRYING`：进入重试
- `FAILED`：失败终止

每个状态记录：
- request_id
- 时间戳
- 耗时（ms）
- 输入摘要
- 命令数
- 失败原因（若有）

---

## 4. 分阶段实施

### Phase 1（本轮落地）
1. 引入请求状态机（最小可用）
2. 增加结构化实时日志（JSON风格）
3. 将关键路径挂到状态机：接收、分析、网络返回、执行

### Phase 2
1. 固化双阶段生成：Plan -> Commands
2. 增强语义质量门（可配置规则）
3. 不通过时自动回喂重试（最多 N 次）

### Phase 3
1. 构建回归测试集（50+复杂图形语句）
2. 建立质量指标：语义正确率、执行成功率、平均延迟
3. 形成发布验收基线

## 6. 验收标准（Phase 1）

1. 每个请求至少产生 4 个状态日志（RECEIVED/ANALYZED/GENERATED/EXECUTED|FAILED）
2. 日志包含 request_id 与耗时字段
3. 不影响现有绘图执行流程
4. 代码无新增 linter 错误

---

## 7. 风险与对策

- 风险：日志过多影响控制台可读性
  - 对策：后续新增 debug 开关与日志级别
- 风险：状态迁移点遗漏
  - 对策：以 request_id 做兜底，对异常统一写 FAILED
