# AutoCADMindAI 开发代理规范

## 工作边界

1. **不破坏现有可用主链路**
   - 主链路包括：用户发送消息 → 模型推理 → 结果展示；停止按钮中断；CAD 执行安全确认与执行。
   - 任何修改不得导致上述链路中断或产生未处理的异常。
   - 新增功能必须通过可回退开关控制，默认关闭，确保主链路不受影响。
2. **新能力采用可回退开关**
   - 所有新增能力（新工具、新知识源、新 UI 组件）必须在配置文件中增加开关（如 `config/features.yaml`）。
   - 开关默认值为 `false`，需显式开启后方可使用。
   - 开关命名规范：`enable_<feature_name>`，便于动态配置。
3. **每次改动必须可验证**
   - 任何代码变更必须附带对应的单元测试或集成测试。
   - 核心模块（`autocad_controller.py`、`ai_model.py`、`core/orchestrator.py`）的测试覆盖率不低于 80%。
   - 提交前运行 `pytest tests/` 确保所有测试通过。

## 执行原则

- **先读现状，再改代码**
  - 修改前必须理解现有模块职责、接口定义、调用链。
  - 优先复用现有类/函数，避免重复实现。
- **小步提交**
  - 每次提交只解决一类问题（如“新增知识库检索开关”、“修复 CAD 执行超时”）。
  - 提交信息格式：`[模块] 简要描述`，如 `[orchestrator] 添加意图识别的可回退开关`。
- **改后自检**
  - 本地运行主程序，手动验证改动相关功能。
  - 检查日志是否有异常堆栈。
  - 确认未引入新警告（`pylint` 分数不下降）。

## 多能力协同策略

### 总体流程

1. 用户输入经 `main_ai_cad.py` 收集。
2. `core/orchestrator.py` 负责意图识别与路由：
   - 优先由大模型进行意图分类（默认）。
   - 若配置中启用了特定工具（如 `enable_kb_retriever`），且命中对应关键词/规则，则触发工具调用。
3. 工具调用结果（如知识库条目、ERP 数据、文件内容）作为上下文，与原问题一起再次交给大模型，生成最终回答。
4. 若工具调用失败，则执行失败兜底策略，保证会话继续。

### 工具调用规范

- **知识库检索（KB）**
  - 通过 `connectors/kb_sqlserver/retriever.py` 的 `KBRetriever.retrieve()` 调用。
  - 超时时间 5 秒，超时后降级为“暂无法检索知识库”。
- **ERP 接口（未来）**
  - 预留 `connectors/erp/` 目录，调用需经过配置开关。
- **文件检索（未来）**
  - 预留 `connectors/file_retriever/` 目录，支持本地文档索引。
- **CAD 执行**
  - 仅当意图明确且用户确认后，通过 `AutoCADController` 执行。
  - 执行前检查 AutoCAD 是否运行，未运行则提示启动。

## 失败兜底

### 通用原则

- 任何工具调用失败时，向用户解释失败原因，并给出下一步建议。
- 不因单个工具失败中断整个会话，确保主链路可用。

### 具体场景

| 工具/模块             | 失败场景          | 兜底行为                                  |
| :---------------- | :------------ | :------------------------------------ |
| KBRetriever       | 连接超时 / 无结果    | 回复：“暂无法检索公司知识库，以下内容基于通用知识。”           |
| AI 模型服务           | API 不可用 / 超时  | 回复：“AI 服务暂时不可用，请检查配置或稍后重试。” 并记录日志。    |
| AutoCADController | AutoCAD 未运行   | 回复：“未检测到 AutoCAD 进程，请启动 AutoCAD 后重试。” |
| AutoCADController | 命令执行失败        | 回复：“执行失败：{错误信息}。请检查命令参数或手动操作。”        |
| IPC 桥接服务          | 端口冲突 / 请求格式错误 | 返回 HTTP 500，并记录详细错误日志。                |

### 日志记录

- 所有工具调用入口和出口必须记录 INFO 级别日志，包含输入参数、耗时、结果状态。
- 错误必须记录 ERROR 级别日志，包含堆栈信息（敏感信息需脱敏）。

## 开发流程规范

1. **需求澄清**
   - 新功能需先明确需求类型、涉及模块、功能描述、技术约束（参照 USER.md 格式）。
2. **代码评审**
   - 所有 PR 必须至少一名其他开发人员审阅。
   - 评审重点：是否遵循工作边界、是否添加回退开关、测试覆盖率是否达标。
3. **版本控制**
   - 主分支为 `main`，禁止直接推送。
   - 功能分支命名：`feature/<描述>`，修复分支：`fix/<描述>`。
4. **文档同步**
   - 修改 `SOUL.md`（AI 行为）或 `AGENTS.md`（开发规范）时，必须同步更新相关 README 或用户手册。

## 附录：可回退开关示例

配置文件位置：`config/features.yaml`（若不存在则创建默认模板）

```yaml
features:
  enable_kb_retriever: false
  enable_erp_connector: false
  enable_file_retriever: false
  enable_auto_cad_execution: true   # 主链路开关，默认开启
```
