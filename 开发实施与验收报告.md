# AutoCADMindAI 开发实施与验收报告（分步落地版）

> 文档定位：本报告是项目从“当前可用版本”走向“企业级智能中枢”的执行基线。后续开发、联调、验收、上线均以本文件为准。  
> 适用范围：**Python GUI/AI能力层/SQL Server 2014/ERP 对接/共享文件检索**（不包含 C# 窗体重构细节）。

---

## 1. 项目总目标与边界

## 1.1 总目标

在不推翻现有架构（AutoCAD C# 插件 + Python AI层）的前提下，实现以下能力：

1. 企业统一知识库问答（不限 CAD）
2. 可控 CAD 指令执行（仅明确 intent=command）
3. AI 记忆持久化（会话记忆 + 文档记忆）
4. AI 人格设定可配置并可版本化
5. 全局配置入库（配置文件丢失可恢复）
6. 金蝶 ERP API 查询
7. 局域网共享文件检索

## 1.2 非目标

- 不一次性重写全部 UI
- 不一次性替换全部模型/供应商
- 不做不可回滚的大版本切换

---

## 2. 当前基线（实施起点）

当前已具备：

- PyQt 主界面与聊天执行链路（`main_ai_cad.py`）
- 模型适配层（`ai_model.py`）
- CAD 控制器（`autocad_controller.py`）
- IPC bridge（`ipc_bridge.py`）
- 设置窗口（`ui/settings_window.py`）

关键安全门已存在：

- 仅当 `intent == command` 且有命令时执行 CAD

---

## 3. 总体实施策略（从无到有）

采用 **“底座优先、能力逐层叠加、每层可独立验收”** 的策略：

- Phase 0：工程基线与规范
- Phase 1：数据库底座（知识 + 配置 + 审计）
- Phase 2：知识库导入与检索（RAG V1）
- Phase 3：意图编排 + AI记忆 + 人格设定
- Phase 4：ERP 与共享文件能力
- Phase 5：稳定性、灰度发布与运维

每个阶段都定义：

- 输入条件（Entry）
- 开发任务（Build）
- 验收标准（Exit）
- 失败回滚（Rollback）

---

## 4. 分阶段实施与验收

## Phase 0：工程基线与治理（1周）

### 目标

统一规范，避免后续多模块并行开发失控。

### 开发任务

1. 新增目录约定：`core/`、`connectors/`、`config/`、`sql/`
2. 统一配置文件结构（保留兼容 `ai_config.json`）
3. 建立日志规范（request_id / session_id / user_id）
4. 建立版本命名和发布标识规则

### 验收标准

- [ ] 目录结构落地，空目录不超过 1 个
- [ ] 日志统一包含 request_id
- [ ] 配置读取优先级文档化（DB > 本地）
- [ ] 新增模块不影响现有发送/停止链路

### 回滚策略

- 保留原入口与原配置读取逻辑开关

---

## Phase 1：数据库底座（2周）

### 目标

完成 SQL Server 2014 核心建库：知识、配置、审计三大域。

### 开发任务

1. 建立知识库表（domain/document/version/chunk/tag/synonym/acl）
2. 建立配置中心表（版本化配置 + 启用状态）
3. 建立管理员口令与审计表
4. 建立任务与工具调用日志表
5. 所有关键表和字段添加扩展属性（MS_Description）

### 验收标准

- [ ] 建表脚本一次执行成功（空库）
- [ ] 核心外键和索引可用
- [ ] 每张核心表至少含表级说明 + 关键字段说明
- [ ] 配置单活约束生效（同一配置仅一个启用）

### 回滚策略

- 使用事务执行 DDL 批次
- 建库脚本附 drop/disable 脚本（仅测试环境）

---

## Phase 2：知识库导入与检索（2~3周）

### 目标

实现“文档入库 -> 可检索 -> 有引用回答”。

### 开发任务

1. `ingest.py`：文档解析、切块、关键词、标签
2. 入库流程：staging -> quality check -> publish
3. `retriever.py`：关键词 + 同义词 +（可选）Full-Text
4. 返回证据：文档名/版本/章节

### 验收标准

- [ ] 导入 10+ 份文档成功率 >= 95%
- [ ] “完整流程类问题”可输出步骤顺序
- [ ] 回答引用覆盖率 >= 95%
- [ ] 无证据问题返回“未找到公司标准依据”

### 回滚策略

- 导入任务按 job_id 可撤销（staging 未发布不入正式）

---

## Phase 3：意图编排 + AI记忆 + 人格设定（2周）

### 目标

将系统从“单轮问答”升级为“多能力编排 + 可持续记忆 + 可控人格”。

### 开发任务

1. `core/intent_router.py`：KB_QA / ERP_QUERY / FILE_SEARCH / CAD_COMMAND / CHAT
2. `core/orchestrator.py`：按意图路由到对应 connector
3. 记忆体系：
   - 会话短期记忆（最近 N 轮）
   - 长期记忆（数据库持久化）
   - 文档记忆（入库后可复用）
4. 人格设定体系（系统提示词模板版本化）：
   - 专业度（严谨/简洁）
   - 语气（中性/正式）
   - 安全规则（禁止无依据断言）

### 验收标准

- [ ] 多轮追问上下文不丢失
- [ ] 人格切换后回答风格可感知且稳定
- [ ] CAD 执行安全门未被破坏
- [ ] 记忆写入/读取有日志可追溯

### 回滚策略

- 人格模板和编排器开关可回退为“直连模型模式”

---

## Phase 4：ERP + 共享文件能力（2~3周）

### 目标

接入金蝶 API 与局域网文件检索，形成业务闭环。

### 开发任务

1. `erp_kingdee/client.py`：token 获取、缓存、刷新、重试
2. 业务 API 封装：物料规格查询（首个场景）
3. `file_share/finder.py`：UNC 实时扫描检索
4. （可选）`file_share/indexer.py`：索引模式

### 验收标准

- [ ] 料号查询闭环：缺参数时能追问
- [ ] token 过期自动刷新成功率 >= 99%
- [ ] 文件检索返回可访问路径（UNC）
- [ ] 工具调用失败可解释、可审计

### 回滚策略

- ERP、File 功能可独立禁用，不影响 KB 与 CAD 基础能力

---

## Phase 5：上线准备与运维（1~2周）

### 目标

达到可灰度、可监控、可诊断的生产状态。

### 开发任务

1. 指标看板：P50/P95 时延、调用成功率、无结果率
2. 灰度发布策略：10% -> 30% -> 全量
3. 回归测试清单固化
4. 诊断导出（日志 + 配置版本 + 最近失败调用）

### 验收标准

- [ ] 关键链路稳定通过 7 天观察
- [ ] 严重错误率低于阈值
- [ ] 灰度用户满意度达标（可定义问卷指标）

## 5. 模块清单与验收用例（逐模块）

## 5.1 模块：配置中心（数据库版本化）

### 规则

- 配置变更必须：`INSERT 新版本 -> 旧版本 status=0`
- 禁止覆盖式 UPDATE
- 同类配置仅一个启用

### 验收用例

1. 保存配置后新版本生成，旧版本禁用
2. 配置回滚可恢复到历史版本
3. 配置按钮未认证密码不可保存
4. 验证失败达到阈值触发锁定

## 5.2 模块：AI 记忆

### 记忆分类

- `short_memory`：会话窗口内短记忆
- `long_memory`：跨会话用户偏好/常用上下文
- `doc_memory`：文档化知识（团队共享）

### 验收用例

1. 同会话连续追问上下文一致
2. 重启后长期记忆仍可读取
3. 文档源文件删除后，知识回答仍存在（来自 DB）

## 5.3 模块：AI 人格设定

### 目标

将“回答风格”与“安全边界”配置化、版本化。

### 验收用例

1. 设为“专业严谨”后答案结构固定
2. 无证据时拒绝编造
3. 人格切换不影响 CAD 执行判定

## 5.4 模块：知识检索

### 验收用例

1. 流程问题返回有序步骤
2. 标准问题返回规范条款+引用
3. 无结果时返回可操作建议（如建议提供关键词）

## 5.5 模块：ERP 查询

### 验收用例

1. 料号缺失追问
2. token 过期自动刷新后查询成功
3. ERP 不可达时优雅失败并记录日志

## 5.6 模块：共享文件检索

### 验收用例

1. 文件存在则返回 UNC 路径
2. 权限不足时给出明确错误
3. 超时可中断并返回部分结果

---

## 6. 质量门槛（全局）

上线前最低要求：

1. 功能完整率 >= 95%
2. 严重缺陷（P0）= 0
3. 回答引用覆盖率 >= 95%
4. 幻觉率（无依据回答）<= 5%
5. 关键请求 P95 响应时间满足目标（可按内网环境设定）

## 7. SQL Server 2014 建表与扩展属性（示例基线）

> 说明：以下为核心表基线脚本（可作为 `sql/01_schema_with_description.sql`）。可在此基础继续扩展业务字段。

```sql
/* =====================================================
   AICAD_KB - Core Schema with Extended Properties
   SQL Server 2014
   ===================================================== */

/* ---------- 1) Domain ---------- */
CREATE TABLE dbo.kb_domain (
    domain_id BIGINT IDENTITY(1,1) PRIMARY KEY,
    domain_code NVARCHAR(50) NOT NULL UNIQUE,
    domain_name NVARCHAR(100) NOT NULL,
    is_active BIT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT GETDATE()

EXEC sys.sp_addextendedproperty @name=N'MS_Description', @value=N'知识领域主表（如 CAD/HR/FIN）',

EXEC sys.sp_addextendedproperty @name=N'MS_Description', @value=N'领域主键ID',
@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_domain',@level2type=N'COLUMN',@level2name=N'domain_id';

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_domain',@level2type=N'COLUMN',@level2name=N'domain_code';

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_domain',@level2type=N'COLUMN',@level2name=N'domain_name';

/* ---------- 2) Document ---------- */
CREATE TABLE dbo.kb_document (
    doc_id BIGINT IDENTITY(1,1) PRIMARY KEY,

EXEC sys.sp_addextendedproperty @name=N'MS_Description', @value=N'知识文档主表',
@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_document';

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_document',@level2type=N'COLUMN',@level2name=N'doc_code';

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_document',@level2type=N'COLUMN',@level2name=N'status';

/* ---------- 3) Document Version ---------- */
CREATE TABLE dbo.kb_document_version (
    version_id BIGINT IDENTITY(1,1) PRIMARY KEY,
    doc_id BIGINT NOT NULL,
    version_no NVARCHAR(50) NOT NULL,
    effective_date DATETIME NOT NULL,
    expire_date DATETIME NULL,
    is_active BIT NOT NULL DEFAULT 1,
    change_summary NVARCHAR(MAX) NULL,
    created_at DATETIME NOT NULL DEFAULT GETDATE(),
    CONSTRAINT FK_kb_ver_doc FOREIGN KEY(doc_id) REFERENCES dbo.kb_document(doc_id)

EXEC sys.sp_addextendedproperty @name=N'MS_Description', @value=N'文档版本表',
@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_document_version';

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_document_version',@level2type=N'COLUMN',@level2name=N'version_no';

/* ---------- 4) Chunk ---------- */
CREATE TABLE dbo.kb_chunk (
    chunk_id BIGINT IDENTITY(1,1) PRIMARY KEY,
    doc_id BIGINT NOT NULL,

@level0type=N'SCHEMA',@level0name=N'dbo',@level1type=N'TABLE',@level1name=N'kb_chunk',@level2type=N'COLUMN',@level2name=N'step_order';
```

## 8. 数据验证与验收 SQL（建议）

```sql
-- 1) 检查是否存在无说明字段（示意：检查表级）
SELECT t.name AS table_name, ep.value AS description
FROM sys.tables t
LEFT JOIN sys.extended_properties ep
  ON ep.major_id = t.object_id
 AND ep.minor_id = 0
 AND ep.name = 'MS_Description'
WHERE t.name IN ('kb_domain','kb_document','kb_document_version','kb_chunk','sys_config_item');

-- 2) 检查配置单活
SELECT config_key, COUNT(*) AS active_count
FROM dbo.sys_config_item
WHERE status = 1
GROUP BY config_key
HAVING COUNT(*) > 1;

-- 3) 检查知识分块是否可用于流程排序
SELECT TOP 20 doc_id, section_title, step_order
FROM dbo.kb_chunk
ORDER BY doc_id, step_order;
```

## 9. 发布控制与变更流程

1. 开发分支完成模块后提交“模块验收记录”
2. 联调环境通过后进入灰度名单
3. 灰度阶段记录关键指标并评审
4. 评审通过后全量发布

每次发布必须附带：

- 变更清单
- 回滚方案
- 验收截图/日志
- 风险说明
